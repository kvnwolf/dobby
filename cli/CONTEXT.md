# cli (`@kvnwolf/dobby`)

The `dobby` CLI: the kit's mechanical execution layer. Run it from a project to
snapshot the working environment (`dobby env`) — worktree, cmux, detected
capabilities, config presence — with every fact resolved locally (no network).

## Files

- `src/index.ts` — the bin adapter (shebang `#!/usr/bin/env bun`). Logic-free EXCEPT two concessions. (1) The **streaming split**: a live `dobby dev` (the `dev` command WITHOUT `--dry-run`) is intercepted here and handed to `lifecycle.runDev` — it manages a concurrent process group (portless-wrapped main + secondaries) with signal forwarding and lives until the group exits, which the synchronous capture seam (`run()`) cannot supervise; everything else — every finite command AND `dev --dry-run` — routes through `run()` unchanged. (2) When `--hook` is in argv it drains `process.stdin` (the PostToolUse payload) and threads it as `run()`'s third argument. Otherwise reads only `process.argv`/`process.cwd()`, calls `run()`, writes stdout/stderr, exits. Not an import target, not unit-tested.
- `src/run.ts` — **the module's interface**, `run(argv, cwd, stdin?)`. Parses argv (`node:util` `parseArgs`, strict), dispatches on the first positional, and formats output (all `key: value` / JSON rendering lives here, incl. the db/dev/up/down dry-run plans — `up`'s plan renders the folded setup phase first — AND the capability-filtered usage/help). Returns the process outcome as data. `stdin` carries the `check --hook` payload; every other command ignores it. For `dev` it PLANS only — `run()` is the capture seam, so it renders the `--dry-run` plan and enforces the no-app gate ('nothing to run'); the LIVE dev group is owned by index.ts's streaming path (which intercepts `dobby dev` before `run()`), so `run()` never spawns a dev server. The usage text is COMPUTED per invocation: `buildUsage(detectCapabilities(cwd))` (the SAME detection over the passed cwd that `env`'s `capabilities:` line uses) lays out `tasks.usageCommands` into the aligned Commands block; the Options block is static. Every usage-printing path (bare, parse error, unknown command) uses this filtered text.
- `src/detect.ts` — internal capability detector. `detectCapabilities(root)` reads `<root>/package.json` (dependencies ∪ devDependencies) and returns the capability names in fixed catalog order. TOLERANT: a missing/unparseable package.json contributes no signals (never throws). Single-package only.
- `src/config.ts` — the SOLE reader of `dobby.config.json`. `loadConfig(root)` is tolerant: `null` when the file is absent, `{ ok: false, error }` when unparseable, `{ ok: true, config }` when valid. The schema `DobbyConfig` (`files[]` + optional `setup[]`/`teardown[]`/`checks[]`) is EXPORTED — `env` observes only presence + parseability, but `check` reads `checks[]` from it (and future commands read the rest).
- `src/runner.ts` — the shared workroot-pinned child-process utility (every command's single spawn seam) AND the single bin resolver. `resolveWorkroot(cwd)` finds the git top-level (null outside a repo — the ONE call that reads the ambient cwd); `requireWorkroot(cwd)` is its fail-hard variant for action commands. `resolveBin(name, { scope, root? })` is THE resolver every dobby TOOL spawn routes through: `scope: "bundled"` (portless, biome, tsc, knip, taze) resolves from DOBBY's OWN tree — walk `node_modules/.bin/<name>` upward from this module's location (`import.meta.url`; handles nested + hoisted layouts), then a `require.resolve` fallback on the shipping package's `bin` field, then the bare name — so bundled tools resolve PATH-INDEPENDENTLY (the field-bug fix: `env` spawned a BARE `portless` off PATH and got `devUrl: null`); `scope: "consumer"` (vite, vitest, drizzle-kit, email) resolves `<root>/node_modules/.bin/<name>` first, else the bare name. In both scopes PATH is only a last-resort fallback. System tools (git, curl, cmux, sh, bunx) stay BARE and never call it. `runCapture`/`runInherit` spawn (synchronously) with `cwd` pinned to the passed `root` (never ambient), returning data (never throwing). `spawnDetached(cmd, args, { root })` is the ASYNC streaming counterpart: it `spawn`s a long-running child DETACHED (its own process group) with inherited stdio, cwd pinned, and returns the live `ChildProcess` so the caller manages the GROUP as a unit (wait for exit, forward signals, `process.kill(-pid)` to tear the whole group down) — consumed by the `dev` executor. `spawnBackground(cmd, args, { root, logPath })` is the NON-supervised background variant: DETACHED, cwd pinned, stdio redirected to `logPath`, `unref`d so the parent exits while the child lives, returning the child's pid — `up`'s plain-terminal start writes it to a pidfile so a later `down` can signal the group. `node:child_process` `spawnSync`/`spawn` only (never Bun.spawn).
- `src/envinfo.ts` — env-snapshot assembly. `collectEnv(root)` returns the `EnvSnapshot` data (cmux, worktree, branch, capabilities, config, devUrl, runPane, browserPane); it reads `CMUX_WORKSPACE_ID` from `process.env`, resolves the workroot ONCE via `runner.ts` and pins every git/portless/cmux spawn to it, resolves devUrl (`portless get <name>` for vite projects, portless resolved BUNDLED via `runner.resolveBin` so it works off PATH) and the kit pane refs (`cmux list-panes` → `list-pane-surfaces`, matched by surface title), and folds in detect/config. Every fact degrades to null/false/[] — it never throws. `resolveDevUrl` and `discoverPanes` are EXPORTED and reused by the `up`/`down` lifecycle (so `up`'s browser-pane / liveness target is the identical URL `env` reports, and `up` reuses / `down` closes the same kit panes `env` discovers).
- `src/tasks.ts` — PURE task inference (no spawning, no fs, no process — the most-tested module). `checkPipeline(capabilities, config, flags)` returns the ORDERED `CheckStep[]` plan for `dobby check`: full gate (no flags) = biome, tsc, knip, then the capability-gated build (needs `vite`) + test (needs `vitest`), then config `checks[]` extras; selective flags (`lint`/`types`/`unused`/`build`/`test`) pick a SUBSET of the tool steps in that same order and drop the extras. Capability gating is decided HERE: a build/test step whose capability is absent still appears in the plan but carries a `skipNote` (the single line `check.ts` prints). `check.ts` executes the plan. `dbTasks(capabilities)` returns the inferred `db:*` map (a `DbTaskSet`: `mode` = none/single + a name→`DbCommand` map): drizzle is the ONLY db tool, so the SHORT names always map to drizzle-kit (`db:push` → `drizzle-kit push`, etc.); no drizzle → an empty set (mode "none"). `UPDATE_ARGS` is the pure `taze --interactive` arg vector. `devPlan(capabilities, config)` returns the ordered `DevPlan` for `dobby dev`: `main` (null with no vite → the no-app gate; else the `.vite` cache-clears + the `vite dev` command that lifecycle wraps in `portless run`) and concurrent `secondaries` (`email dev --dir src/emails` for react-email — the canonical emails dir, the only secondary). `config` is threaded for forward-compat (v1 has no config-driven dev behavior). `lifecycle.ts` executes them. `usageCommands(capabilities)` returns the capability-FILTERED `dobby` help Commands list (`UsageCommand[]` — name + one-line description): the universal set (env/check/update) always, dev/up/down ONLY with `vite` (`up` described as the prepare + run entry point), and the `db:*` entries ONLY with a db capability — each carrying its ACTUAL resolved name (from `dbTasks`) and the concrete shell command it runs. `run.ts` renders it (the field-report fix — the help no longer advertises commands the repo can't run).
- `src/check.ts` — the quality-gate EXECUTOR of the `tasks.ts` plan. `check(files, cwd, flags, fix?)` shells out to the tools (never an in-process/WASM API) and returns findings + notes as DATA (`run.ts` formats). `fix` (the `--fix` flag) applies biome's SAFE fixes FIRST — project-wide `biome check --write .` (reaching the config files too, not just `src/`), or over just the named files on the per-file path — so the pre-commit gate never fails on formatting the edit hook did not reach; the UNSAFE rewrites (e.g. `==`→`===`) are NEVER applied. It then runs the selected pipeline and reports whatever biome could NOT safely fix. `checkHook(stdin, cwd)` is the edit-time PostToolUse path: it parses the hook payload's `tool_input.file_path` defensively, runs `biome check --write` (SAFE fixes) on that one file so formatting is fixed in place, and returns `{ surface: false }` for every guard (unparsable payload / no file_path / missing file / not a git repo / no `dobby.config.json` marker / file outside the workroot / extension outside `ts,tsx,js,jsx,json,jsonc,css` / biome could not run / biome fixed everything) or `{ surface: true, groups }` when unfixable findings remain (run.ts prints them to STDERR and exits 2). Per-file (file args): biome ONLY over those files — the edit-adjacent fast path, no pipeline/flags. Project-wide (no file args): runs the `checkPipeline` plan — `biome check --reporter=json .` + `tsc --noEmit --pretty false` + `knip --reporter json`, plus (when the gating capability is present) `vite build` / `vitest run --reporter=json`, then config `checks[]` shell extras (`sh -c`, run last, fail-fast among themselves). BUNDLED bins (biome/tsc/knip) resolve from DOBBY's OWN node_modules; the capability-gated vite/vitest bins resolve from the CONSUMER's node_modules (a bundled second vite would clash with the consumer's plugins). Bin lookup reads each package's `bin` field (`binFrom`); knip's `exports` blocks the `./package.json` subpath, so its root is found by walking up from the resolved main entry (`pkgRootFrom`). knip is TOLERANT — a run that can't start (no package.json) folds to zero findings, never a hard error. Biome findings count only `error`/`warning` severities; tsc lines parse via `path(line,col): error TSxxxx: message`. ALL selected steps run (report everything); the report's `exitCode` is the FIRST failing step's code. Returns `{ ok: false, error }` (not a git repo / a BUNDLED tool unresolvable) or `{ ok: true, groups, notes, exitCode }`.
- `src/lifecycle.ts` — the action-command executors (the inferred `db:*` tasks, `update`, `dev`, `up`/`down` — `up` now owns the folded setup phase). The IMPURE counterpart to the pure planners in `tasks.ts`: it resolves the workroot, touches the filesystem, and spawns children (always through `runner.ts`, cwd pinned to the workroot), returning DATA (`DbTaskReport`/`UpdateReport`/`DevReport`/`UpReport`/`DownReport`) that `run.ts` renders. `planDev(cwd)` is the `dev` CAPTURE path (used by `run()` for `--dry-run` and the no-app gate): detect capabilities, `devPlan`, and fold "no app main" into the hard "nothing to run" error — no spawn. `runDev(cwd)` is the `dev` STREAMING path (used ONLY by index.ts): the native `.vite` cache-clear (`rmSync`), then the portless-wrapped main (portless resolved BUNDLED via `runner.resolveBin`, run via `process.execPath`) + the consumer-local secondaries spawned as ONE managed group (`runManagedGroup`) — each child DETACHED via `runner.spawnDetached`, and ANY child exit or a SIGINT/SIGTERM to dobby tears the whole group down (`killGroup` → `process.kill(-pid)`), resolving with the MAIN's exit code. `planDev` RESOLVES every bin (consumer app/secondary bins + the bundled portless) via `runner.resolveBin` into a `ResolvedDevPlan`, so the `--dry-run` render and the real spawn read the SAME resolved paths (part c: the resolution is observable in the plan). The streaming spawn + kill mechanics are NOT CI-tested (they spawn real servers) — covered by the wrap-stage human smoke + the verifier's live recipe. `buildSetupPlan(root, config)` builds the ordered SETUP-PHASE plan (a `SetupAction[]`) `up` runs first — (1) `bun install`, (2) `.worktreeinclude` re-materialization (LINKED-worktree only), (3) config `setup[]` extras — and `executeSetup(plan, root)` runs it fail-fast (returning the first failing step's exit code + a `failure` note). A LINKED worktree is detected by `git rev-parse --path-format=absolute --git-dir` ≠ `--git-common-dir`; the MAIN checkout root is the parent of the common `.git` dir, where `.worktreeinclude` is read; each pattern is matched against the main checkout (literal fast-path; simple `*`/`?` globs walk the tree skipping `.git`/`node_modules`), and any match MISSING at the worktree is copied over (idempotent — never overwriting a present file). `DOBBY_SKIP_INSTALL=1` is a documented TEST SEAM that skips only `bun install`. (There is no standalone `setup` command — it was folded into `up` per Findings #32.) `runDbTask(name, cwd, { dryRun })` detects capabilities from `cwd`, resolves `name` through `dbTasks`, and resolves the tool bin CONSUMER-local via `runner.resolveBin` (`<root>/node_modules/.bin/<tool>`, bare fallback) — `--dry-run` returns that RESOLVED bin + args for `run.ts` to print (part c), a real run spawns it (inherit stdio). No db capability, or an unknown name, is a hard error whose message already lists the available (short) drizzle task names. A detected-but-not-installed tool (resolveBin fell back to the bare name) degrades to a `run dobby up` failure note (exit 127), not a raw ENOENT. `runUpdate(cwd)` resolves taze BUNDLED via `runner.resolveBin` and runs `taze --interactive` with inherited stdio; fails hard outside a git repo. `runUp(cwd, { dryRun })` / `runDown(cwd, { dryRun })` are the run-lifecycle pair. `up` is THE single lifecycle entry point: after the git precondition it runs the SETUP PHASE (`buildSetupPlan` → `executeSetup`, fail-fast — any failure exits nonzero and the run phase never starts), THEN the run phase. Both commands resolve their run-phase decisions (capabilities, devUrl, cmux, neon creds from `.env.local` — `up` via `resolveUpContext`) into an ordered `UpAction[]` / `DownAction[]` plan that `run.ts` renders for `--dry-run` (`up`'s `UpPlan` carries the `setup` phase + `runSkipped` reason alongside the run `actions`) and `executeUp` / `executeDown` walks for a real run. `up` is liveness-first + idempotent: no vite → a graceful exit-0 no-op (`no app to run`, reached only AFTER the setup phase); neon capability with either `NEON_API_KEY`/`NEON_PROJECT_ID` missing → hard exit 1 (no main-DB fallback); else probe (`curl -sf --max-time 5 <devUrl>`, alive → ensure panes, exit 0), provision the `dobby/<slug>` neon branch (`bunx neonctl branches create` idempotently, rewriting `.env.local` DATABASE_URL* from the connection strings), start (cmux → browser pane `--direction right` + run terminal `new-split down --surface` targeted, `cd <workroot> && bunx dobby dev`; else `spawnBackground` a detached `bunx dobby dev` with pid/log under `.dobby/` and a `.gitignore` append), then a liveness wait (6×5s, `sleepSync` via `Atomics.wait`). `down` teardown, best-effort: close discovered kit panes (`cmux close-surface`), kill the pidfile's process group (SIGTERM; a stale pid → remove the file silently), delete the neon branch (missing → ok), run config `teardown[]` extras (`sh -c`). The cmux orchestration (`createPanes`, with a `captureSurfaceRef` + discovery-diff fallback since the cmux stdout ref format is runtime-unverified), the neon branch create/delete, the detached spawn, and the liveness probe/wait are NOT CI-tested — covered by the verifier's live recipe + the wrap-stage human smoke.
- `src/run.test.ts` + `__fixtures__/` — the co-located vitest suite (run via `vitest`, discovered from the repo root by vitest's default globs) and its hand-written sample projects. Tests call `run()` in-process with a fixture path (or a throwaway temp git repo) as `cwd`; they never import `detect.ts`/`config.ts`/`envinfo.ts`/`check.ts` directly. The `check` integration slices build a throwaway git repo (inline biome.jsonc + tsconfig + hand-written lint/type errors) and run the REAL biome + tsc.
- `tsconfig.base.json` + `biome/core.jsonc` + `biome/react.jsonc` — the exported PRESETS (thin-file model). `tsconfig.base.json` is the strict bundler base consumers extend (`strict`, `noUncheckedIndexedAccess`, `noEmit`, `skipLibCheck`, `isolatedModules`, `esModuleInterop`, `resolveJsonModule`, `module: preserve`, `moduleResolution: bundler`). `biome/core.jsonc` extends `ultracite/biome/core`; `biome/react.jsonc` extends both `ultracite/biome/core` and `ultracite/biome/react`. These preset files are themselves biome-formatted config assets — the repo's own root `biome.jsonc` (extending `@kvnwolf/dobby/biome/core`) governs the whole tree; there is no `vite.config.ts`.
- `package.json` — bin `dobby` → `./src/index.ts`, `type: module`, and the preset `exports` (`./tsconfig` → `./tsconfig.base.json`, `./biome/core` → `./biome/core.jsonc`, `./biome/react` → `./biome/react.jsonc`). The bundled toolchain (`@biomejs/biome`, `ultracite`, `typescript`, `knip`, `taze`, `portless`) sits in **runtime `dependencies`** so consumers inherit the tool bins + preset resolution transitively.
- `README.md` — the **npm package front page** (consumer-facing, English): what `@kvnwolf/dobby` is (zero-config toolchain + env-aware run lifecycle), the single-devDep install, the thin-config `extends` model, the FULL command reference with per-command examples + flags, the canonical conventions (`src/emails`, `.env.local` NEON creds), the `dobby.config.json` schema, and the per-capability inferred defaults. It documents the WHOLE surface and states that the live `dobby` help is capability-filtered (shows only the applicable subset). The repo-root `README.md` covers the plugin/kit; this one is the CLI's.

## Interface

`run(argv: string[], cwd: string, stdin?: string): Promise<{ exitCode: number; stdout: string; stderr: string }>`

- bare (no positional) → the capability-filtered usage on stdout, exit 0. The Commands list is COMPUTED from the cwd's detected capabilities (`usageCommands`): universal commands (env/check/update) always, dev/up/down only with `vite` (`up` is the prepare + run entry point), db:* only with a db capability (the ACTUAL resolved short drizzle names). The field-report fix: the old static help advertised dev/up/down/db:* in repos with neither capability.
- `--version` / `-v` → package version + newline on stdout, exit 0.
- `env` → the environment snapshot on stdout, **always exit 0** (env NEVER fails; unresolvable facts are null). Six fields:
  - `cmux` — the `CMUX_WORKSPACE_ID` value, or `null`.
  - `worktree` — the enclosing git worktree root (git's resolved path), or `null` outside a repo.
  - `branch` — the current git branch, or `null` outside a repo / on detached HEAD.
  - `capabilities` — the detected capabilities as a comma-separated list, or `none`.
  - `config` — `true`/`false`: whether a parseable `dobby.config.json` exists at the root.
  - `devUrl` — the portless-resolved dev URL for a vite project (`portless get <name>`, name = the package.json `portless` key or the `@scope/`-stripped package name), or `null` (no vite capability / portless missing / errors). Network-free — a local CLI resolve, never a liveness probe.
  - `runPane` / `browserPane` — the kit pane surface refs, discovered through cmux IPC when `CMUX_WORKSPACE_ID` is set (surfaces titled `dobby-run-<slug>` / `dobby-browser-<slug>`, slug = workroot basename), else `null`.
  - `env --json` → the same facts as one JSON object: string|null scalars, `capabilities` as a JSON array, `config` as a JSON boolean.
- `check [file...] [--fix] [--lint|--types|--unused|--build|--test]` → the quality gate. No file args = the project-wide pipeline (`tasks.ts`): full gate (no flags) = biome, tsc, knip, build-if-vite, test-if-vitest, + config `checks[]` extras; ANY selective flag = run ONLY the flagged tool steps (extras excluded). File args = biome-only fast path over those files (selective flags ignored). `--fix` applies biome's SAFE fixes FIRST — project-wide `biome check --write .` (reaching the config files too), or over just the named files on the per-file path — then runs the selected pipeline and reports whatever remains; it composes with the selective flags (`--fix --lint` = fix then lint-report) and with the no-flag full gate, and NEVER applies biome's unsafe rewrites (`==`→`===`). **This is the pre-commit standard: run `dobby check --fix` before committing — dobby ships no `commit` command, so whoever commits (a human, a script, the `/dobby:commit` skill) runs the gate first.** Token-lean stdout: one `file:line message` line per finding, grouped + labelled per tool, capped at 50 per tool (`…N more` tail); then each step NOTE on its own line — a capability skip (`build: skipped (no vite capability)`) or a build/test/extra failure. A capability-gated step whose capability is absent is skipped with that single note (a skip is NOT a failure). ALL selected steps run (report everything, no stop-at-first-tool); exit = the FIRST failing step's code (findings ⇒ 1) or 0 when clean, and 1 on a hard error (outside a git repo / a bundled tool unresolvable — message on stderr).
- `check --hook` → the edit-time PostToolUse mode. Reads the hook payload from **stdin** (`{ tool_name, tool_input: { file_path }, ... }`), applies biome's SAFE auto-fixes to the edited file IN PLACE, and surfaces ONLY unfixable findings: exit **2** with the token-lean findings on **STDERR** (the channel Claude Code shows the model on exit 2 — never stdout). Every guard is a **silent exit 0** (empty stdout AND stderr): unparsable payload / no `file_path` / missing file / not a git repo / no `dobby.config.json` marker at the workroot / file outside the workroot / extension outside `ts,tsx,js,jsx,json,jsonc,css` / biome could not run / biome fixed everything. Harness noise must never block an edit. Invoked by `plugin/hooks/hooks.json` through a double-guarded LOCAL-bin shell one-liner (config marker + executable bin present, else exit 0; never `bunx`).
- `dev [--dry-run]` → run the app. The composition (`tasks.ts` `devPlan`): the MAIN — a `.vite` cache-clear (`rm -rf node_modules/.vite`, admin's preamble now inferred for every vite dev) followed by the vite dev server WRAPPED in `portless run` (portless from DOBBY's bundled tree, the vite bin consumer-local; portless wraps ONLY the main) — then the concurrent SECONDARIES (`email dev --dir src/emails` for react-email, the only secondary, consumer-local, un-wrapped). `--dry-run` prints the full ordered plan (`Dev plan (dry-run):` + one shell-style line per command, cache-clear → portless main → secondaries) through the CAPTURE seam WITHOUT spawning — the portless wrapper and the consumer bins render as their RESOLVED absolute paths (bare tool name when a consumer bin is absent). No app (no vite) → exit 1 with 'nothing to run' (`up` is the graceful path). A LIVE `dobby dev` (no `--dry-run`) is dispatched by the bin's streaming path, NOT `run()`: main + secondaries spawn as ONE managed process group (cwd pinned to the workroot) that a child exit or SIGINT/SIGTERM tears down whole, exiting with the main's code. Fails hard outside a git repo.
- `db:<task> [--dry-run]` → an inferred database task. The name resolves through the capability-driven `db:*` map (`tasks.ts` `dbTasks`): drizzle is the one db tool, so the SHORT names always map to drizzle-kit (`db:generate/migrate/push/check/studio`). `--dry-run` prints the RESOLVED command (`db task (dry-run):` + `<resolved-consumer-bin> <args>` + the pinned `cwd:`, the bin an absolute `node_modules/.bin` path or the bare name when absent) WITHOUT spawning; a real run resolves the tool bin CONSUMER-local and inherits stdio (finite runs stream; `db:studio` is interactive). Unknown name → exit 1, message listing the available names; no db capability → exit 1. A detected-but-not-installed tool → exit 127 with a `run dobby up` note.
- `update` → `taze --interactive` from DOBBY's OWN bundled taze, inheriting stdio (the picker is user-driven and terminates with them). Fails hard outside a git repo. Not auto-invoked in tests (interactive) — covered by usage-text presence + the verifier's live recipe.
- `up [--dry-run]` → THE single lifecycle entry point: **prepare + run the workspace (idempotent)**, mechanizing worktree creation + execute Step 2. **(0)** fail hard outside a git repo (the git precondition wins over every gate). **(1)** SETUP PHASE (the folded former `setup` command): `bun install` (always — the inferred default; `DOBBY_SKIP_INSTALL=1` skips only this), then `.worktreeinclude` re-materialization (LINKED-worktree only — copy each main-only match MISSING at the worktree, idempotent), then config `setup[]` extras (`sh -c`, sequential, FAIL-FAST) — any setup-phase failure exits nonzero and **the run phase never starts**. **(2)** no vite → a graceful exit-0 no-op (`no app to run`), reached only AFTER the setup phase. **(3)** RUN PHASE (unchanged): a neon project missing either cred in `.env.local` → hard exit 1 (guaranteed branch isolation, NO silent main-DB fallback); otherwise a single liveness probe (already up → ensure the kit panes exist, exit 0); the `dobby/<slug>` neon branch (`bunx neonctl branches create --name … --project-id … --output json`, idempotent, then rewrite `.env.local` DATABASE_URL / DATABASE_URL_UNPOOLED from the branch connection strings); the START — under cmux the positional pane layout (browser pane RIGHT of Claude via `new-pane --type browser --direction right`, run terminal BELOW it via a surface-targeted `new-split down --surface <browser-ref>`, panes named `dobby-browser-<slug>` / `dobby-run-<slug>`, `cmux send "cd <workroot> && bunx dobby dev"`), else a detached `bunx dobby dev` (pid → `.dobby/dev.pid`, stdio → `.dobby/dev.log`, `.dobby/` gitignored); then a liveness retry wait (6×5s) — never reachable → exit 1 with the `portless trust` hint. `--dry-run` renders the FULL ordered plan (`Up plan (dry-run):` + the setup-phase lines THEN the run-phase lines, or the `no app to run` skip line) WITHOUT installing / probing / spawning / touching cmux or neon.
- `down [--dry-run]` → tear the run down (best-effort), mechanizing finish teardown: close the discovered kit panes (`cmux close-surface`), kill the `.dobby/dev.pid` process group (SIGTERM; a stale pid → remove the file silently), delete the `dobby/<slug>` neon branch (`bunx neonctl branches delete`, missing → ok), run config `teardown[]` extras (`sh -c`, sequential). Nothing to clean → exit 0 no-op. `--dry-run` renders the plan (`Down plan (dry-run):`, `(nothing to clean)` when empty). Fails hard outside a git repo.
- unknown command / malformed flag → error + the SAME capability-filtered usage on stderr, exit 1 (a no-capability repo's unknown-command usage hides dev/up/down/db:*; a drizzle-only repo's shows the short db names). Unknown commands additionally carry the version-skew hint line `if this command is expected, run \`bun update @kvnwolf/dobby\`` as the second line.

The bin is the only reader of `process.cwd()`; every other caller (tests, future subcommands) passes `cwd` explicitly.

## Invariants

- **Runtime-portable core**: `run.ts`, `detect.ts`, `config.ts`, `envinfo.ts`, `runner.ts` use `node:*` imports (plus the plain JSON import) only — no `Bun.*` globals, no `bun:` modules — because vitest imports them at Node/Vite runtime while Bun runs them in production. `runner.ts` uses `node:child_process` `spawnSync` (never `execFileSync`, so a nonzero child exit is data not a throw; never `Bun.spawn`).
- **Signals check `dependencies ∪ devDependencies` only** — NEVER `peerDependencies`; missing dependency fields are treated as empty objects.
- **Signal matching forms**: exact dependency name (`react` never matches `react-email`), OR-of-several (`drizzle` ← `drizzle-orm`|`drizzle-kit`), and scope prefix (`react-email` ← `@react-email/*`). Fixed catalog order: `vite`, `tanstack-start`, `react`, `neon`, `drizzle`, `react-email`, `vitest`, `expo`.
- **`env` never fails**: `collectEnv` degrades every unresolvable fact to `null`/`false`/`[]`, and formatting cannot throw — the exit code is always 0. (Action commands added in later tasks fail hard; `env` is the snapshot exception.)
- **`config.ts` is the ONLY reader of `dobby.config.json`** — tolerant tri-state (missing → null / unparseable → error object / valid → config); no other module opens that file.
- **The detectors are pure over their `root` argument** — they never read `process.cwd()`; `run()` owns all output formatting (the modules return data, never rendered lines).
- **Workroot pinning is THE runner invariant**: every child process spawns through `runner.ts` with `cwd` pinned to the resolved workroot (`git rev-parse --show-toplevel`), NEVER the ambient cwd — killing the run-in-the-wrong-checkout bug class. `resolveWorkroot` is the sole exception (it runs git FROM the ambient cwd, because that is how the workroot is discovered). Action commands outside a repo fail hard (`requireWorkroot`); `env` is exempt and degrades to null. `up`'s setup phase consumes the fail-hard + `runInherit` foundation (`bun install` + config `setup[]` extras).
- **One bin resolver, owning-tree-first, PATH last (`runner.resolveBin`).** Every dobby TOOL spawn routes through it; system tools (git, curl, cmux, sh, bunx — neonctl via bunx) stay bare and never do. BUNDLED tools (portless, biome, tsc, knip, taze) ALWAYS prefer dobby's OWN dependency tree (the whole point of bundling is zero reliance on globals) — resolved PATH-INDEPENDENTLY by walking `node_modules/.bin` up from the runner module's own location, never a `which`-style PATH lookup. CONSUMER tools (vite, vitest, drizzle-kit, email) ALWAYS prefer the consumer's own `node_modules/.bin` (never a global that may version-skew). PATH is a last-resort fallback in BOTH scopes, never the first choice — this is the field-bug fix (a bare `portless` off PATH made `env` print `devUrl: null`). The `--dry-run` plans for dev/db RENDER the RESOLVED absolute paths so the resolution is observable, and the real spawn reads the SAME resolution. `check.ts` keeps its own equivalent bundled/consumer resolution (`binFrom`/`pkgRootFrom`/`resolveConsumerBin`, run via `process.execPath`) — an already-correct site with a real-tool integration test, deliberately not rerouted to avoid regressing it.
- **`up`'s setup-phase re-materialization is gated on being a LINKED worktree, not on `.worktreeinclude` presence** — a plain checkout carrying the file copies NOTHING (the gate is `--git-dir` ≠ `--git-common-dir`), and the copy fills MISSING files only, so a second `up` (or a locally-edited worktree file) is never clobbered.
- **`env` stays network-free**: devUrl and pane refs are resolved through LOCAL CLI/IPC only (`portless get`, `cmux list-*`) — no liveness probe, no HTTP. Any tool being missing/erroring folds the fact to null, and `env` still exits 0.
- **Capabilities read the caller's cwd, not the workroot** — a single-package project runs dobby at its root, and the task-1 fixtures (which live inside this repo) rely on capability detection reading their own `package.json` rather than this repo's top-level.
- **The usage/help is capability-filtered over the PASSED cwd, never a workroot resolve** — `buildUsage` calls `detectCapabilities(cwd)` (the same detection `env` uses), so the fixtures that live inside this git repo report their OWN capabilities; a workroot resolve would collapse every fixture to this repo's (empty) set. Universal commands (env/check/update) are never filtered out; dev/up/down gate on `vite`; db:* gates on a db capability and shows the resolved names from `dbTasks`. The Options block stays static (a flags reference).
- **`check` shells out — never an in-process tool API**: biome + tsc run as child processes with machine reporters (`biome --reporter=json`; parse `tsc --pretty false` text — TS7 has NO JS API), never the Biome WASM js-api. The tool binaries resolve from DOBBY's OWN dependency tree (`createRequire(import.meta.url)` → the package's `bin` field), NEVER the consumer's — consumers install nothing. The CONSUMER's `biome.jsonc` / `tsconfig.json` are the live configs (thin-file model: they extend `@kvnwolf/dobby/tsconfig` and `/biome/*`), discovered by each tool walking up from the pinned workroot.
- **Biome findings count only `error`/`warning`** — `info`/`hint` diagnostics (e.g. the `recommended`-field deprecation notice) are surfaced by biome but MUST NOT fail the gate; the finding filter drops them so a clean project exits 0.
- **The check PLAN is pure (`tasks.ts`), the EXECUTION impure (`check.ts`)** — `checkPipeline` never spawns; all child processes stay in `check.ts` through `runner.ts`. build/test bins resolve from the CONSUMER's tree (never dobby's — dual-vite would break the consumer's plugins); biome/tsc/knip from DOBBY's. A capability-absent build/test step is SKIPPED with a single note, not a failure. Every SELECTED step runs regardless of earlier failures (report everything); config `checks[]` extras are the one exception — fail-fast among themselves. The exit code is the FIRST failing step's; selective flags subset the pipeline and exclude the extras.
- **Streaming split: `run()` is the CAPTURE seam, the bin owns the live stream.** A long-running `dobby dev` (a concurrent group with signal forwarding) cannot flow through the synchronous `run()` seam, so index.ts intercepts a live `dobby dev` (no `--dry-run`) and hands it to `runDev`; `run()` handles only the `dev` `--dry-run` plan and the no-app gate — it NEVER spawns a dev server. `--dry-run` on a streaming command therefore always routes through the capture path (testable). The dev COMPOSITION is pure (`tasks.ts` `devPlan`) and fully covered by dry-run plan assertions; only the spawn/kill EXECUTION (`lifecycle.runDev` / `runManagedGroup`) is impure and out of CI reach. `portless` wraps ONLY the main process (never the secondaries); the `.vite` cache-clear is part of the inferred vite dev; the managed group is DETACHED-spawned via `runner.spawnDetached` (cwd pinned) and torn down as a unit with `process.kill(-pid)` (`node:child_process`, no `concurrently` dep).
- **`db:*` inference: drizzle-only → short names always.** drizzle is the ONE db tool (`dbTasks`, pure): its capability exposes the SHORT `db:*` names (`db:generate/migrate/push/check/studio`), each mapping to drizzle-kit; no drizzle → mode "none" and every `db:*` name is a hard error. The drizzle-kit bin resolves CONSUMER-local (never dobby's bundled tree). `update`'s taze resolves from DOBBY's OWN tree (bundled).

- **`up`/`down` safety invariants (load-bearing, must not soften).** Neon isolation is guaranteed: a neon-capability project with EITHER `NEON_API_KEY` / `NEON_PROJECT_ID` missing from `.env.local` makes `up` FAIL HARD — there is NO silent fallback to the shared main database. The cmux pane layout is POSITIONALLY exact and focus-independent: the run terminal is created by `new-split down --surface <browser-ref>` (targeting the browser surface by ref, never by focus); 50/50 sizes are accepted (cmux has no size-on-split, and it persists manual divider adjustments). slug = workroot basename; the neon branch is `dobby/<slug>` (SLASH), the panes are `dobby-{browser,run}-<slug>` (DASHES). Both commands are ACTION commands — fail hard outside a git repo (the git precondition wins over EVERY gate — `up`'s setup phase, its no-app gate, and its run phase). `up`'s setup phase runs BEFORE the no-app gate and the run phase, and a setup-phase failure short-circuits both (the run phase never starts). The `--dry-run` plan and the real execution both derive from ONE resolved decision set so a dry run never lies about what a real run would do.

## What's intentionally NOT here

- **Deep cwd-pinning, exercised end-to-end only for extras** — `up`'s setup phase consumes `requireWorkroot` (fail-hard outside a repo) and `runInherit` (config `setup[]` extras + `bun install`), and its tests observe extras landing at the workroot (marker files) — but `bun install` itself is never spawned in tests (the `DOBBY_SKIP_INSTALL=1` seam, which still skips only the install inside `up`), so the install spawn's cwd pinning is covered by the later live recipes, not CI.
- **cmux listing stdout format is runtime-unverified** — the pane discovery parser is deliberately tolerant (scan lines for `<kind>:<ref>` tokens, substring-match titles); live cmux pane behavior is covered by the wrap-stage human smoke, not CI (no reachable cmux surface here).
- **Workspace / monorepo detection** — v1 is single-package only (interview decision); `env` reports a flat capability list. The former workspace-grouping machinery was removed with the `capabilities` command. Per-workspace inference is a future arc.
- **The build/test INCLUDE path is untested here** — the capability-gated build (`vite build`) and test (`vitest run`) steps are exercised in tests ONLY on the capability-SKIP path (fixtures carry no vite/vitest, and running them is forbidden by the task). "build/test actually run when the capability is present" is covered by the verifier's live recipe, not CI.
- **`env`, `check` (incl. `--hook` and `--fix`), the inferred `db:*` tasks, `update`, `dev`, and the `up`/`down` run-lifecycle pair all exist now.** There is NO standalone `setup` command — it was folded into `up`'s setup phase (STATE.md Findings #32): the user always wants a workspace running when opening it, so `up` prepares AND runs. `setup` therefore hits the ordinary unknown-command path (exit 1 + the `bun update @kvnwolf/dobby` hint), exactly like the removed `commit` command. There is likewise NO `commit` command — it was removed (STATE.md Finding #31): `dobby commit` wrapped git for little value, so the cross-project standard is now running `dobby check --fix` before committing, and the git/gh ceremony lives in the `/dobby:commit` SKILL, not the CLI. The streaming split (part a) landed with `dev`: the bin owns the live streaming path, `run()` stays the capture seam. The `dev` STREAMING execution (inherited-stdio group spawn, the `process.kill(-pid)` group teardown, the SIGINT/SIGTERM forwarding) is NOT CI-tested — the composition is fully covered through `devPlan`'s `--dry-run` plan assertions, but the spawn/kill mechanics need a real server (forbidden in tests), so they are left to the wrap-stage human smoke + the verifier's live recipe, exactly as `update` and `db:studio` are handled.
- **`up`/`down` real execution is out of CI reach** — every branch is asserted through the `--dry-run` PLAN (cmux env-var set/unset, neon fixtures with/without `.env.local` creds, teardown extras) plus the early-return paths run for real (up's no-app gate, both fail-hard-outside-git, down's nothing-to-clean no-op, down's stale-pidfile cleanup, a down `teardown[]` extra's file side-effect). The IMPURE work — the curl liveness probe/wait, `bunx neonctl` branch create/delete + the DATABASE_URL* rewrite, the cmux pane orchestration (with the runtime surface-ref capture + discovery-diff fallback), the detached `dobby dev` spawn + `.gitignore` append, cmux pane REUSE/CLOSE — needs a live server / cmux / neonctl and is covered by the verifier's live recipe + the wrap-stage human smoke, NOT CI. down's pane-close and up's liveness curl target are the specific gaps the plan cannot show (down discovers panes live → empty in CI; devUrl is null without portless).
- **`db:*` real execution + `update` are not spawned in tests** — the db:* mapping is asserted only through `--dry-run` (no spawn), and a real `db:*` run in a fixture (no tool installed) is asserted to FAIL, never to succeed; `db:studio` and `update` are interactive (inherit stdio, terminate with the user — the streaming-dispatch prefactor lands in the later `dev` task), covered by the verifier's live recipe not CI. A `db:*` command in a project with NO db capability exits 1 with a "no database capability detected" message (a thin-spec gap — no test constrains the wording).
- **Config schema surfacing** — `env` reports only whether a config is present + parseable; of the `files/setup/teardown/checks` fields, `check` reads `checks[]`, `up`'s setup phase reads `setup[]`, and `down` reads `teardown[]`. `files[]` is skill-consumed (doc-sync), never by the CLI.
- **`up`'s setup-phase glob support is untested** — the `.worktreeinclude` matcher supports simple `*`/`?` globs (Conductor `file_include_globs` parity) via a bounded tree walk, but the fixed tests exercise only the literal-filename fast path (`.env.local`); the glob branch is covered by real-world use, not CI.
