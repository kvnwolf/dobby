# Domain glossary

The vocabulary of the dobby kit. Use these terms exactly — in skills, agents, docs, and conversation.

- **Execution host** — WHERE a work session runs. There is a SINGLE host: the **terminal host** (see its own entry). The KIT owns the worktree + run lifecycle; the coordinator and verifier reach the running app via the devUrl `dobby env` resolves (portless-based, distinct per worktree) plus a curl liveness check — not by reading a run-script terminal — and drive the UI via the cmux-browser → claude-in-chrome → curl ladder. (Conductor, the former second host, was removed this session; everything it did — workspace-as-worktree, `auto_run_after_setup`, `.conductor/settings.toml`, `file_include_globs` — is captured for a possible future re-add in the ADR that supersedes ADR-0005, landing at `/dobby:wrap`.)
- **Terminal host** — the kit's single execution host: a plain `claude` session (incl. ssh, and inside cmux), where the KIT owns the worktree + run lifecycle. `/dobby:scope` creates+enters the per-goal worktree (native `EnterWorktree`, branch `worktree-<slug>` under `.claude/worktrees/`) and brings it up via `bunx dobby up` (setup phase + run), which `/dobby:execute` Step 2 re-runs idempotently, and `/dobby:finish` tears it all down post-merge via `bunx dobby down`. Environment mechanics are delegated to the `@kvnwolf/dobby` CLI throughout. See also **cmux enrichment**.
- **cmux enrichment** — the optional layer that activates when `CMUX_WORKSPACE_ID` is present (auto-set in every cmux pane; cmux = the manaflow-ai native macOS terminal). `bunx dobby up` creates the kit panes — a named run pane (`dobby-run-<slug>`) and a named browser pane (`dobby-browser-<slug>`) — and `bunx dobby env` surfaces their refs (`runPane`/`browserPane`, rediscovered by title); the cmux-browser CLI is the top rung of the verifier's UI ladder. A plain terminal (ssh/tmux, no `CMUX_WORKSPACE_ID`) degrades gracefully: `dobby up` starts the run as a detached background job (no panes), and the UI ladder skips to claude-in-chrome/curl.
- **Run script** — the command that starts the app. It is NO LONGER a config key: `dobby dev` INFERS it from the project's detected capabilities (zero-config à la Vercel; see **Task inference**) — the main dev server wrapped in **portless**, plus any capability-detected secondaries (e.g. react-email) run concurrently — and `dobby up` owns starting it. The `portless get` identity is UNCHANGED: the dev URL is obtained deterministically via `portless get <name>` — worktree-branch-prefixed (portless's own behavior, so distinct per worktree), resolved never parsed from terminal output — and `dobby env` reports it as `devUrl`; liveness is a curl health-check against that URL. A project with no app (a lib/CLI/plugin like dobby itself) infers no dev command, so `devUrl` is null, `up`/`down` no-op the run cleanly, and the verifier verifies programmatically.
- **up / down** — the run-lifecycle pair the `dobby` CLI exposes, mechanizing `/dobby:scope`'s bring-up, `/dobby:execute` Step 2, and `/dobby:finish` teardown. **`dobby up`** prepares AND runs the workspace in one idempotent, liveness-first command: a **setup phase** (`bun install` + `.worktreeinclude` re-materialization + any `setup[]` config extras — worktree prep is folded into `up`, not a separate command), then it starts the app only if not already live (under cmux it creates the kit panes — browser + a dev-terminal running `dobby dev`; on a plain terminal it spawns a detached process with a pidfile) and waits for liveness; a no-app project finishes the setup phase and reports 'no app to run'. For a neon-capability project it provisions a per-worktree branch and rewrites `.env.local` (missing creds → hard fail, never a silent fallback to the shared DB). **`dobby down`** tears it back down, best-effort: closes the kit panes / kills the detached process, deletes the neon branch, and runs `teardown[]` extras.
- **Architect** — the main thread. It frames the work, decides, and reviews what comes back. It NEVER writes code; that asymmetry (expensive cognition at the top, hands-on work delegated) is the kit's core design.
- **Worker** — one of the five custom agents that do the hands-on work: `researcher`, `implementor`, `reviewer`, `verifier`, and `test-author`. Each has a fixed role, toolset, model, and effort.
- **Work session** — one end-to-end run over a single goal, moving through stages: scope → interview → research → spec → execute → wrap.
- **Stage** — one step of a work session. Each stage is a skill that does its job and ends by handing the user the next command to type — handoffs are typed, never auto-invoked, so each stage runs on its own declared model/effort.
- **Finish** — the post-merge teardown stage (`/dobby:finish`): a manual command run after the PR is merged. It checks the PR is MERGED (confirming before destroying anything otherwise), runs **`bunx dobby down`** (which closes the kit-opened cmux panes / kills the detached run, deletes the per-worktree Neon branch, and runs the config's `teardown[]` extras), removes the worktree + branch (native `ExitWorktree` when this session created it, raw-git fallback for orphans), and `git pull`s the main checkout.
- **One session per goal** — the invariant `/dobby:scope` enforces on the terminal host: each session/pane owns ONE goal plus its kit-created worktree on a goal-named branch. It is NOT "one worktree per machine" — MULTIPLE goals run in PARALLEL worktrees, one per cmux pane/session (the cmux value-prop). Scope guards only NESTING (a session already inside a worktree must start the next goal in a new pane, since the native tool can't nest) and slug collisions — it does NOT refuse merely because other sessions' worktrees exist in `.claude/worktrees/`. Teardown is per goal via `/dobby:finish`.
- **Convention skill** — a skill that encodes the user's standard application stack (TanStack Start + Drizzle/Neon + Better Auth, the `@/shared` form/data system) rather than methodology: `data-processing`, `data-fetching`, and `module-conventions`. It auto-activates while building, intentionally references the consuming project's module file conventions (deep-path imports and the role-based file taxonomy — no barrels), and is NOT a work-session stage.
- **STATE.md** — the ephemeral session doc at the target repo's root, created by `/dobby:scope`. Accumulates `## Exploration`, `## Findings (interview)`, `## Research`, `## Spec`, `## Work log`. Disposed at wrap — it is never committed.
- **`docs/` durable artifacts** — where the kit's skills write knowledge that OUTLIVES a work session (the durable counterpart to the ephemeral `STATE.md`). All under the project's `docs/` tree, committed, created lazily (never an empty dir): the **decision-map** at `docs/maps/` (`/dobby:map`'s dependency-linked investigation tickets for one planning effort), the **out-of-scope KB** at `docs/out-of-scope/` (`/dobby:triage`'s rejected enhancement requests, one concept file each), and the **discarded-frictions KB** at `docs/learn-discarded/` (`/dobby:learn`'s field frictions deliberately NOT turned into skill edits, one concept file each). ADRs stay in `docs/adr/` (project-owned, standard convention); `STATE.md` stays ephemeral at the repo root.
- **dobby.config.json** — the kit-owned per-project contract at the repo ROOT (next.config.js style — not `.claude/`, which is the host's reserved namespace). JSON, shrunk to `files[]` (doc-sync `update_when` triggers) plus OPTIONAL `setup[]` / `teardown[]` / `checks[]` extras that run AFTER (in addition to) the capability-inferred defaults, and an optional `tracker` key (the per-project issue backend — see **tracker**) — the old `run` key and any task list are GONE (inferred now; see **Task inference**). The `dobby` CLI is its sole reader of the mechanical keys: `check` folds in `checks[]`, `up` runs `setup[]` (in its setup phase), `down` runs `teardown[]`. `files[]` stays skill-consumed (`/dobby:commit`, `/dobby:resolve-conflicts`); the `tracker` key is read by the tracker-touching skills (`/dobby:scope`, `/dobby:backlog`, `/dobby:triage`, `/dobby:commit`) via `backlog/references/trackers.md`. Its presence at the root is ALSO the "dobby project" marker the edit hook guards on. Written by `/dobby:onboard`; existing repos migrate via `/dobby:migrate-config` (clean cut from the legacy `.claude/commit.config.yml` + vite-plus). A no-app project (a lib/CLI/plugin like dobby itself) infers no dev command, so `devUrl` stays null.
- **tracker** — the per-project issue backend the work skills read and write, selected by the optional `tracker` key in **dobby.config.json**: **GitHub Issues** by default (the `gh`-authenticated repo; key absent or `type: "github"`), **Linear** (`type: "linear"` + a `team` key like `VON`), or **local** (`type: "local"`, a plain `BACKLOG.md`). GitHub keeps a natural `repo → tracker` derivation (zero-config); Linear needs the `team` key because it has none. _Avoid_: issue tracker, ticket system.
- **Build loop** — the per-task loop that turns a spec into verified code, one SEPARATE agent per role, looping until review and verify both pass. It is **3 OR 4 steps**: implement → review → verify by default, gaining a leading `test-author` step (write → implement → review → verify) only when the repo has a test suite AND the spec marked the task test-first. The name is deliberately count-agnostic. Encoded once in the `dobby:execute` skill's `references/build-workflow.md` and reused by `/dobby:dispatch`.
- **Test-author** — the worker (`dobby:test-author`) that runs the conditional first step of the build loop: it writes a task's tests **from the spec alone, never seeing the implementation**, producing the fixed contract the implementor must satisfy. Blindness to the code is what makes the tests anti-tautological — an independent source of truth. It runs ONCE per task (outer-loop retries re-implement against the SAME tests); the reviewer judges the tests' static quality and the verifier runs them.
- **Wave** — a batch of area-disjoint tasks the build-loop workflow runs in parallel. Overlapping areas serialize; shared-state verification never overlaps.
- **Work log** — the per-task implementation record (diff summary, decisions, deviations) an implementor RETURNS (it never writes `STATE.md`); the architect appends entries serially as the single writer.
- **Verify recipe** — the per-task steps a verifier executes against the already-running app to prove the task works. Written at spec time, consumed at execute time.
- **needs-human** — terminal task status when review or verify never passed within the retry caps. The workflow doesn't thrash; it escalates.
- **Dispatch** — the lightweight ad-hoc path: a scoped task handed to one worker (or the single-task build loop), no `STATE.md`, no waves.
- **Prototype** — throwaway code that answers ONE design question, then dies. Two branches: **logic** (a minimal TUI over a pure, portable module) and **UI** (3-5 radically different variants on one route with a floating switcher).
- **Namespacing** — inside the plugin, every cross-reference is fully qualified: `/dobby:<skill>` for skills, `dobby:<agent>` for `subagent_type`/`agentType`. Bare names are reserved for things OUTSIDE the plugin (`deep-research`, `find-docs`, built-in `Plan`/`Explore`).
- **Session indicator** — the copy-pasteable pointer to a recorded Claude Code session that `/dobby:mark` emits (transcript `.jsonl` path + repo, worktree root, the `STATE.md` path, the `/dobby:*` skills it invoked, and a note). Consumed by `/dobby:learn`, which digests that session to improve a kit skill from how it actually behaved in the field. Together `mark` (capture, in the consumer project) and `learn` (digest + edit, in this repo) are the kit's **self-improvement loop**; they couple to host paths (`~/.claude/projects`, `CLAUDE_CODE_SESSION_ID`) on purpose.
- **Teach** — the kit's on-demand pedagogy skill (`/dobby:teach`): the architect teaches the user ONE topic in-session — explains it from trusted resources, runs a tight feedback loop to verify understanding, and records the demonstrated understanding as evidence. It is a light, conversational architect capability (teaching is interaction, not code work, so no worker is dispatched) and is NOT a work-session stage — it's invoked on demand. Distinct from the `mark`/`learn` self-improvement loop, which evolves the kit rather than the user.
- **Capability** — a detected fact about a project, derived from signals in its dependencies/marker files (e.g. `vite`, `tanstack-start`, `react`, `neon`, `drizzle`, `react-email`, `vitest`, `expo`). Capabilities DRIVE the mechanical layer — `dobby` infers each task from them (the check pipeline, the `dev` composition, the `db:*` set, the capability-filtered help); see **Task inference**. `dobby env` reports the detected list; a capability is triggered by one or more **detection signals**.
- **Detection signal** — the concrete dependency or file that triggers a **capability** (e.g. `@tanstack/react-start` in deps → `tanstack-start`, `drizzle-orm`/`drizzle-kit` → `drizzle`); lives in code/tests, never printed.
- **Task inference** — the zero-config mapping (à la Vercel) from a project's detected **capabilities** to the commands `dobby` runs, REPLACING per-project config/scripts (no `run` key, no `package.json#scripts`, no vite-plus task table). Inferred: the check pipeline (Biome + tsc + knip, plus build-if-vite / test-if-vitest), the `dev` composition (portless-wrapped main → concurrent secondaries like react-email), and the `db:*` task set (drizzle short names). Project-specific residue that resists inference lives in `dobby.config.json`'s `setup[]` / `teardown[]` / `checks[]` extras.
- **Workroot** — the pinned working directory every `dobby` child process runs in: the git top-level, resolved ONCE per invocation (`git rev-parse --show-toplevel`) and pinned as the explicit `cwd` on every spawned tool, detached process, and cmux pane command (`cd <workroot> && …`, since cmux panes take no `--cwd`). A load-bearing invariant — it kills the run-in-the-wrong-checkout bug class. Action commands (`check`, `up`, `down`, `dev`, `db:*`, `update`) fail hard outside a git repo; only `env` degrades to null there.
