# Writing agent briefs

An agent brief is the structured comment posted on an issue or PR when it's ready to be actioned. It is the authoritative specification the downstream worker (or `/dobby:scope` session) works from. The original body and discussion are context — the brief is the contract.

The brief states **what should be done**. For an issue that's the change built from nothing; for an external PR it's what's left to do *to the existing diff* — finish it, close gaps, address review points. Same principles either way; the PR example below shows the difference.

## Principles

### Durability over precision

The brief may sit unactioned for days or weeks, and the codebase will move underneath it. Write it so it stays useful even as files are renamed, moved, or refactored.

- **Do** describe interfaces, types, and behavioral contracts.
- **Do** name specific types, function signatures, or config shapes the worker should look for or modify.
- **Don't** reference file paths — they go stale.
- **Don't** reference line numbers.
- **Don't** assume the current implementation structure will remain the same.

### Behavioral, not procedural

Describe **what** the system should do, not **how** to implement it. The worker explores the codebase fresh and makes its own implementation decisions.

- **Good:** "The `SkillConfig` type should accept an optional `schedule` field of type `CronExpression`."
- **Bad:** "Open the skill types file and add a schedule field on line 42."
- **Good:** "When a user runs the command with no arguments, they see a summary of items needing attention."
- **Bad:** "Add a switch statement in the main handler function."

### Complete acceptance criteria

The worker needs to know when it's done. Every brief has concrete, testable acceptance criteria, each independently verifiable.

- **Good:** "Listing items filtered to the pending state returns only those that have been through initial classification."
- **Bad:** "It should work correctly."

### Explicit scope boundaries

State what is out of scope. This stops the worker from gold-plating or making assumptions about adjacent features.

## Template

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** bug / enhancement
**Summary:** one-line description of what needs to happen

**Current behavior:**
Describe what happens now. For bugs, the broken behavior.
For enhancements, the status quo the feature builds on.
For a PR, the state of the diff (what works, what's missing).

**Desired behavior:**
Describe what should happen once the work is complete.
Be specific about edge cases and error conditions.

**Key interfaces:**
- `TypeName` — what needs to change and why
- `functionName()` return type — what it returns now vs. what it should
- Config shape — any new configuration options needed

**Acceptance criteria:**
- [ ] Specific, testable criterion 1
- [ ] Specific, testable criterion 2
- [ ] Specific, testable criterion 3

**Out of scope:**
- Thing that should NOT be changed or addressed here
- Adjacent feature that might seem related but is separate
```

The AI disclaimer is the first line because this brief is posted as a tracker comment.

A completed brief can be handed straight to `/dobby:scope` as the goal for a work session — the Category/Summary/Current/Desired framing is exactly the shape `/dobby:scope` expects.

## Examples

### Good agent brief (bug)

```markdown
## Agent Brief

**Category:** bug
**Summary:** Skill description truncation drops mid-word, producing broken output

**Current behavior:**
When a skill description exceeds the character cap, it is truncated at exactly
the cap regardless of word boundaries. This produces descriptions that end
mid-word (e.g. "Use when the user wants to confi").

**Desired behavior:**
Truncation should break at the last word boundary before the cap and append
"..." to signal truncation.

**Key interfaces:**
- The skill-metadata `description` field — no type change, but the
  validation/processing logic that populates it must respect word boundaries.
- Any function that reads SKILL.md frontmatter and extracts the description.

**Acceptance criteria:**
- [ ] Descriptions under the cap are unchanged
- [ ] Descriptions over the cap are truncated at the last word boundary before it
- [ ] Truncated descriptions end with "..."
- [ ] The total length including "..." does not exceed the cap

**Out of scope:**
- Changing the character cap itself
- Multi-line description support
```

### Good agent brief (enhancement)

```markdown
## Agent Brief

**Category:** enhancement
**Summary:** Add out-of-scope KB support for tracking rejected feature requests

**Current behavior:**
When a feature request is rejected, the issue is closed with a comment. There
is no persistent record of the decision or reasoning; future similar requests
require the maintainer to recall or search for the prior discussion.

**Desired behavior:**
Rejected feature requests are documented in per-concept records that capture the
decision, reasoning, and links to every issue that requested the feature. During
triage, these records are checked for concept matches against incoming requests.

**Key interfaces:**
- The out-of-scope record format — each record has a concept heading, a reason
  section, and a "Prior requests" list of issue links.
- The triage flow reads all out-of-scope records early and matches incoming
  requests against them by concept similarity.

**Acceptance criteria:**
- [ ] Rejecting an enhancement creates or updates a per-concept record
- [ ] The record includes the reasoning and a link to the closed issue
- [ ] A matching record gets the new issue appended to "Prior requests" rather than a duplicate
- [ ] During triage, existing records are checked and surfaced on a concept match

**Out of scope:**
- Automated matching (a human confirms the match)
- Reopening previously rejected features
- Bug reports (only enhancement rejections are recorded)
```

### Good agent brief (PR)

For a PR, "Current behavior" describes the state of the diff, and the brief asks the worker to finish or fix it rather than build from scratch.

```markdown
## Agent Brief

**Category:** enhancement
**Summary:** Finish the contributor's `--json` output flag for the list command

**Current behavior:**
The PR adds a `--json` flag that serializes the item list to JSON. The happy
path works and matches the project's command structure. Two gaps remain: errors
are still printed as human text (not JSON), and the flag has no test coverage.

**Desired behavior:**
With `--json`, all output — including errors — is well-formed JSON on stdout,
and exit codes are unchanged. Human-readable output is untouched when the flag
is absent.

**Key interfaces:**
- The command's error path emits a `{ "error": string }` shape under `--json`
  instead of plain-text error.
- Reuse the serializer the PR already added; don't introduce a second.

**Acceptance criteria:**
- [ ] The command with `--json` emits valid JSON for both success and error cases
- [ ] Exit codes match the non-JSON command
- [ ] A test covers the `--json` success output and one error case
- [ ] Default (non-JSON) output is unchanged

**Out of scope:**
- Adding `--json` to any other command
- Changing the JSON shape of the success payload the PR already defined
```

### Bad agent brief

```markdown
## Agent Brief

**Summary:** Fix the triage bug

**What to do:**
The triage thing is broken. Look at the main file and fix it. The function
around line 150 has the issue.

**Files to change:**
- the triage handler (line 150)
- the types file (line 42)
```

Why it's bad: no category · vague description ("the triage thing is broken") · references file paths and line numbers that go stale · no acceptance criteria · no scope boundaries · no current-vs-desired behavior.
