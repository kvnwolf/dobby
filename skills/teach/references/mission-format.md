# Mission format

The **mission** is the *reason* the user is learning this topic. It grounds every teaching decision — what to teach next, which resources to surface, which retrieval task to design should all trace back to it. Teaching that isn't tied to a concrete goal feels abstract, and you have no way to judge what to teach next.

In the light on-demand flow the mission is usually just held in the conversation — one line, stated back and confirmed. Write it down only when the user wants to keep learning across sessions and asks for something durable. When you do, use this shape.

## Template

```md
# Mission: {Topic}

## Why
{1-3 sentences. The concrete real-world outcome the user is chasing — what changes in their work or life once they have this. Avoid abstract framings like "to understand X"; push for the underlying goal.}

## Success looks like
- {A specific, observable thing the user will be able to do}
- {Another specific thing}

## Constraints
- {Time, budget, prior commitments, learning preferences — whatever bounds the approach}

## Out of scope
- {Adjacent topics the user explicitly does NOT want to chase now — this protects the zone of proximal development}
```

## Rules

- **One mission per topic.** Two unrelated goals are two missions, not one crowded file.
- **Concrete over abstract.** "Ship a Rust CLI to my team by October" beats "learn Rust." "Run a half marathon" beats "get fitter." A concrete mission is one you can measure the next lesson against.
- **Push back on vagueness.** If the user can't say *why*, interview them before teaching anything — a bad mission is worse than no mission. This is the one place teaching pauses to ask.
- **Revise when reality shifts.** Missions change as understanding grows. When the goal moves, update the file and record the shift as a learning record — don't let a stale mission steer future sessions. Confirm with the user before changing it.
- **Keep it short.** If the mission runs past a screen it has stopped being a compass and started being a plan.
